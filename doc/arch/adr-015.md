# ADR-015: Data-driven pre-cast pipeline replacing conditional cascade

## Status

Accepted

## Context

The pre-cast flow (`initiate-cast-with-mode`, game.cljs:243–351) is a ~110-line conditional cascade with 7 branches evaluating casting prerequisites in order: flashback → discard-specific cost → X cost → predetermined target → targeting → mana allocation → direct cast. Each new pre-cast concern adds a branch to this function.

The post-cast flow (resolution-time interactive effects) uses multimethod dispatch — new interactive effect types register via `defmethod` without modifying any coordinating function. The two halves of the selection system use structurally different extension models: conditional cascade (pre-cast) vs multimethod dispatch (post-cast).

Adding a new pre-cast concern (e.g., for Convoke, Offering, Delve, Kicker, Entwine) requires modifying the cascade function. The card pool expansion roadmap targets ~100-200 cards, and several Premodern-era mechanics introduce new pre-cast patterns.

## Decision

We will replace the conditional cascade in `initiate-cast-with-mode` with a data-driven pipeline of pre-cast steps evaluated in order. Each step is a multimethod dispatching on a step keyword. The step ordering is declared as an ordered vector of keywords. `initiate-cast-with-mode` becomes a loop over the pipeline.

Each step evaluates whether it applies and returns:
- `nil` — not applicable, skip to next step
- A selection map — needs player input, pause for selection
- Updated db — step modified game state, continue to next step

New pre-cast concerns register a `defmethod` and add their keyword to the pipeline — same extension model as post-cast selections.

Step-to-step state (e.g., X cost resolution feeding into mana cost calculation) passes through the accumulating db, which is the natural carrier for computed state in an immutable reduction.

## Consequences

- Symmetric extension model: both pre-cast and post-cast use multimethod dispatch
- Adding a new pre-cast concern adds a `defmethod` and a pipeline keyword, not a branch in a 110-line function
- `initiate-cast-with-mode` shrinks from ~110 lines to a pipeline loop (~15 lines)
- Each pre-cast step is independently testable
- The pipeline ordering is explicit data — misordering is visible in the vector, not hidden in branch placement
- Step dependencies (X cost before mana allocation) are enforced by pipeline ordering, not by code flow
- The casting flow is distributed across multiple defmethods rather than readable in one function — tracing the full flow requires following the pipeline vector
- Card definitions are unaffected — this is orchestration-internal refactoring
