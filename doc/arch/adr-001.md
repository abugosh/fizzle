# ADR-001: Pure engine boundary as architectural invariant

## Status

Accepted

## Context

Fizzle's game engine (20 modules in `engine/`) is currently pure functions with zero re-frame dependencies. The event coordination layer (`events/`) wraps these pure functions with re-frame dispatch, side effects, and lifecycle management. This separation exists in the codebase today but is not explicitly declared as an architectural boundary — it could erode as the codebase grows.

The engine's purity enables: testability without re-frame test infrastructure, reuse of game logic in non-UI contexts (bots, tactics evaluation), and reasoning about game rules as pure state transformations.

## Decision

We will treat the boundary between the Interpretation Core (engine/) and Game Orchestration (events/) as an explicit architectural invariant. The Interpretation Core must remain pure functions with zero re-frame dependencies. Game Orchestration is a Manager layer that coordinates calls into the pure engine and handles side effects.

## Consequences

- Engine purity is preserved as the codebase grows — new contributors know this is intentional
- Engine modules remain independently testable with pure function calls
- Game logic can be reused by bots, tactics evaluation, and future non-UI consumers without re-frame
- Selection mechanism logic that is pure (builders, state transforms) belongs in the Engine; re-frame event handlers that wrap it belong in Game Orchestration
- This boundary has interface cost: Game Orchestration must translate between re-frame event/effect model and pure function calls. This cost is accepted because the purity guarantee is worth more than the convenience of mixing concerns.
