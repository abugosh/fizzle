# ADR-003: Subscriptions call Interpretation Core directly for derived state

## Status

Accepted

## Context

Subscriptions (Presentation boundary) need two kinds of data: raw game state from Datascript (Data Foundation) and computed/derived state like "can this card be cast?" or "sort these cards for display" (Interpretation Core).

The natural instinct is to route all reads through a unified Data Foundation interface. However, the Interpretation Core already depends on Data Foundation for its own Datascript queries. If Data Foundation also depended on the Interpretation Core to provide derived state, the result would be a circular dependency: `queries <-> engine`.

Subscriptions sit at the point in the dependency graph where they can import from both Data Foundation and Interpretation Core independently, without creating a cycle:

```
subs ──→ engine   (derived state: can-cast?, sorting, land-card?)
subs ──→ queries  (raw data: get-hand, get-objects-in-zone, get-mana-pool)
engine ──→ queries  (engine internals use queries)
```

An intermediate module (e.g., `db/derived`) was considered but rejected — it would relocate the same coupling without eliminating it, adding indirection for no structural benefit.

## Decision

We will accept that subscriptions import directly from Interpretation Core for game rule queries (`rules/can-cast?`, `rules/can-play-land?`, `rules/land-card?`) and display sorting (`sorting/sort-cards`, `sorting/group-by-type`). The architecture graph will show an explicit Presentation → Interpretation Core edge.

Data access functions that are currently in engine modules but are pure Datascript queries (e.g., `stack/get-all-stack-items`, `grants/get-grants`) will be relocated to `db/queries` so that Data Foundation remains the single interface for raw data access.

## Consequences

- The dependency cycle is avoided — no module depends on both directions
- Subscriptions remain the natural "join point" where raw data and computed state meet
- The Presentation → Interpretation Core edge is declared in the graph, making the coupling visible
- Data Foundation stays focused on Datascript access with no business logic
- Engine functions used by subscriptions must remain pure (no side effects, no re-frame) — this is already guaranteed by ADR-001
- Views that currently import engine directly (phase-bar, opening-hand, battlefield) are acceptable under the same rationale — they are part of the Presentation boundary
