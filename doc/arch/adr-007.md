# ADR-007: Accept implicit value-set contracts for zones, effect types, and selection types

## Status

Accepted

## Context

Four value-sets are shared across components without central registries:

- **Zones** (6 values: `:hand`, `:library`, `:graveyard`, `:stack`, `:battlefield`, `:exile`) ‚Äî referenced in 15+ locations across Card Pool, Interpretation Core, Game Orchestration, and Presentation
- **Effect types** (16+ values: `:add-mana`, `:draw`, `:mill`, `:tutor`, etc.) ‚Äî defined in Card Pool EDN, dispatched by Interpretation Core multimethods
- **Selection types** (12+ values: `:tutor`, `:scry`, `:discard`, etc.) ‚Äî dispatched by Game Orchestration multimethods
- **Stack-item types** (5 values: `:spell`, `:storm-copy`, `:activated-ability`, `:storm`, `:default`) ‚Äî dispatched by Interpretation Core multimethod

The alternative ‚Äî central registries with startup completeness validation ‚Äî would add coordination overhead and run counter to Clojure's open multimethod model where new dispatch values can be added anywhere without modifying existing code.

## Decision

We will keep value-sets implicit, relying on multimethod dispatch as the extension mechanism. Clojure multimethods are inherently open ‚Äî new methods are added by registering defmethods, not by modifying a central registry. This aligns with the project's data-driven extension model where cards are EDN data interpreted by multimethods.

Bare requires in Game Orchestration (events/game.cljs) ensure multimethod registration at namespace load. This convention is accepted as the registration mechanism.

## Consequences

- Adding a new effect type requires adding a defmethod in the appropriate module ‚Äî no registry to update
- No compile-time enforcement that all value-set members have corresponding defmethods; unhandled values hit `:default` or throw at runtime
- Bare requires in events/game.cljs are load-bearing ‚Äî removing one causes silent dispatch failure
- Adding a new zone requires grep+replace across 15+ locations (zone set has been stable at 6 values)
- The multimethod extension model scales with developer discipline; each new card implementation follows the existing pattern of adding defmethods
- Effect and selection type sets grow naturally with new card implementations without coordination overhead
