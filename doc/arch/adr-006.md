# ADR-006: Accept implicit schema attribute contracts

## Status

Accepted

## Context

All components reference Datascript schema attribute keywords (`:object/id`, `:object/zone`, `:card/effects`, `:player/mana-pool`, etc.) directly in code. These keywords are defined in Data Foundation's schema but consumed across 9+ modules in all 6 components without explicit imports or wrapper functions. Renaming an attribute keyword requires changes across all consuming modules with no compile-time detection.

The alternative — encapsulating attribute access behind Data Foundation functions — would add indirection over keyword lookups, running counter to ClojureScript's data-driven idiom where keywords ARE the interface and maps ARE the data transfer format.

## Decision

We will keep schema attribute keywords as implicit vocabulary shared across components. Keywords are values in Clojure's data model, not implementation details to be hidden. This aligns with the project's core principle: "data over code."

Data Foundation remains the single source of truth for schema definition. The implicit contract is accepted because Data Foundation is the lowest-volatility component in the system — schema changes are infrequent by design.

## Consequences

- Attribute renames (e.g., `:object/id` to `:entity/id`) require grep+replace across 9+ modules with no compiler assistance
- This cost is accepted because schema volatility is lowest in the system and renames are rare
- No DTO translation layer or attribute accessor functions needed — data flows as-is across boundaries
- New attributes added to the schema are immediately usable by all components without interface changes
- Contributors must understand the schema vocabulary to work in any component — the schema is load-bearing shared knowledge
