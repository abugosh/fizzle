# ADR-011: Place localStorage persistence under Data Foundation

## Status

Accepted

## Context

`storage.cljs` sits at the project root with no declared component home. It provides localStorage persistence for setup presets, phase stops, and last-used preset. Game Orchestration (events/setup.cljs, events/game.cljs) imports it directly. The module is not represented in the architecture model.

Data Foundation's volatility axis is "storage technology changes" — it encapsulates how data is stored. Currently it covers only Datascript (in-memory Datalog). localStorage is a second storage technology serving a different purpose (persisting user preferences across sessions) but the same axis (if persistence technology changes, these functions change).

## Decision

We will place `storage.cljs` under Data Foundation. Data Foundation encapsulates all storage technology — Datascript for game state, localStorage for user preferences. Moving `storage.cljs` into `db/` (e.g., `db/storage.cljs` or `db/local-storage.cljs`) and adding a Game Orchestration → Data Foundation edge note for preference persistence makes the relationship explicit.

Game Orchestration already depends on Data Foundation (declared edge: "data queries"). Preference persistence flows through the same edge.

## Consequences

- The architecture model is complete — no unmodeled modules
- Data Foundation owns all persistence technology, consistent with its volatility axis
- Game Orchestration's import of storage is covered by the existing Game Orchestration → Data Foundation edge
- If localStorage is replaced (IndexedDB, cloud sync), the change is encapsulated in Data Foundation
- Data Foundation gains a second storage technology (localStorage alongside Datascript), but the two are independent and do not interact
