# ADR-014: Unify modal spell selection under the selection system

## Status

Accepted

## Context

Two parallel mechanisms exist for collecting player input during casting:

1. **Standard selection** (`:game/pending-selection`) — full framework with builder → toggle → confirm → execute multimethod pattern, data-driven validation, 6 domain modules, 19 selection types.
2. **Modal spell selection** (`:game/pending-spell-mode-selection`) — separate code path in game.cljs (~20 lines) that shows mode choices and dispatches a separate event.

Both mechanisms serve the same purpose: present choices to the player, collect input, proceed with casting. The modal mechanism is simpler in the "easy" sense (fewer lines, familiar pattern), but having two mechanisms for the same concern is complection in the Hickey sense — two things braided with the same purpose.

Modal selection already evaluates per-mode targeting validity in `get-valid-spell-modes` (game.cljs:354–366), logic that mirrors the selection system's target validation. As modal grows in complexity (per-mode targeting, per-mode cost previews), maintaining a parallel mechanism means rebuilding selection features in the modal path.

## Decision

We will unify modal spell selection under the standard selection system as a data-driven selection type. Modal becomes `:selection/type :spell-mode` with mode objects as candidates. The selection framework's existing machinery (toggle, auto-confirm, validate) handles the interaction. Views render based on candidate data — if candidates are modes, show mode descriptions; if candidates are cards, show card images.

The difference between modal and card selection is data (what candidates look like, how many to select), not mechanism (builder → toggle → confirm → execute).

`:game/pending-spell-mode-selection` is eliminated. All player input flows through `:game/pending-selection`.

## Consequences

- Single mechanism for all player input collection — one state key, one rendering path, one confirm/cancel cycle
- Modal gets cancel support, validation, and auto-confirm for free from the existing framework
- Per-mode targeting evaluation moves from game.cljs to the spell-mode selection builder
- Views need to handle mode-type candidates (render mode descriptions) alongside card-type candidates (render card images) — the selection type drives the rendering
- The `::select-spell-mode` event handler is eliminated; modal flows through `::toggle-selection` → `::confirm-selection`
- Testing: modal goes from testing a simple event handler to testing the selection multimethod flow, but gains the same production-path test helpers used by all other selection types
- Card definitions are unaffected — `:card/modes` stays the same; only orchestration changes
