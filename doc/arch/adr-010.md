# ADR-010: Route all Card Pool access through Interpretation Core

## Status

Accepted

## Context

Presentation (subs/setup.cljs) imports directly from Card Pool (cards/iggy-pop.cljs) to build a card lookup map for the setup screen. The architecture model declares only one consumer of Card Pool: Interpretation Core. This undeclared Presentation → Card Pool edge bypasses the intended access path.

Card Pool is expected to expand significantly as more card definitions are added. With direct imports, each new consumer would couple to the Card Pool data format. Routing through Interpretation Core provides a stable query interface that can evolve independently of how card data is stored.

## Decision

We will route all Card Pool data access through Interpretation Core. Presentation and other consumers will query card data via Interpretation Core functions (e.g., card name lookups, card grouping, card-by-id resolution), not by directly importing card definition namespaces.

Interpretation Core already depends on Card Pool and interprets card EDN. Adding query functions for card metadata (names, types, sets) is a natural extension of its existing responsibility.

## Consequences

- Card Pool has a single consumer (Interpretation Core) — changes to card data format are isolated
- Presentation accesses card data uniformly through the same path it uses for game rules
- subs/setup.cljs replaces the direct cards/iggy-pop import with Interpretation Core query calls
- Interpretation Core gains card metadata query functions (card-by-id, all-card-names, etc.)
- As Card Pool grows (new sets, new card definitions), the query interface evolves in one place
- Small indirection cost for what is currently a simple lookup, accepted because the interface will serve multiple consumers
