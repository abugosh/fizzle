# ADR-005: All application state lives in re-frame app-db

## Status

Accepted

## Context

Three Reagent atoms in `core.cljs` (`stack-collapsed?`, `gy-collapsed?`, `history-collapsed?`) manage UI panel collapse state outside the re-frame app-db. They work correctly and don't affect game logic, but they break the invariant that all application state flows through re-frame's single-atom model.

The history system only snapshots `:game/db` (the Datascript database), so UI state in app-db under `:ui/` keys would not be affected by fork/replay. The concern that moving state into app-db would complicate history is unfounded.

## Decision

We will move all UI state into re-frame app-db. No mutable state (atoms, vars) should exist outside the re-frame db for any application concern, whether game state or UI chrome. Component-local Reagent atoms in form-2 components (e.g., inline edit state, local collapse toggles scoped to a single component instance) are acceptable — they are not shared state.

## Consequences

- All shared application state is inspectable via re-frame devtools
- The single-atom model is consistent — no exceptions for "it's just UI state"
- Small amount of boilerplate (event handlers, subscriptions) replaces direct atom access
- Future UI state additions follow the same pattern instead of reaching for atoms
- Component-local atoms remain acceptable for truly local, ephemeral interaction state (form inputs, hover tracking, etc.)
