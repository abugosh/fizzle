# ADR-004: Decouple Game Orchestration from history management

## Status

Accepted

## Context

Game Orchestration (`events/game.cljs`) imports Practice Platform's history modules (`history/core`, `history/descriptions`) to manage history entries during complex game events. This creates an import-level dependency from Game Orchestration to Practice Platform, reversing the architectural dependency direction declared in the graph (Practice Platform depends on Game Orchestration).

The direct history calls exist in two places:

1. **`start-turn`** — creates 1-2 history entries (opponent draw + turn start) because two game state transitions happen in a single event. This is a workaround for the lack of proper opponent turns.

2. **`resolve-all-handler`** — loops through stack items, creating one history entry per resolution. This bypasses the global interceptor because a single event produces multiple state transitions. The handler duplicates the interceptor's fork/append logic and must be kept in sync manually.

Both cases share the same root cause: a single re-frame event producing multiple game state transitions that each need independent history entries. The global history interceptor handles the normal case (one event → one entry) but cannot handle batch operations.

## Decision

We will eliminate the direct history coupling by fixing the two underlying design issues rather than adding an intermediate abstraction:

1. **Opponent turns** — Model opponent turns as real turns with active-player switching (`fizzle-gvs7`). Each phase of the opponent's turn becomes its own event, going through the interceptor naturally. The `opponent-draw` hack in `::start-turn` and its manual history entry are eliminated.

2. **Resolve-all as recursive dispatch** — Replace the internal loop in `resolve-all-handler` with a recursive `reg-event-fx` that resolves one item per dispatch, then re-dispatches itself. Each dispatch goes through the interceptor and gets its own history entry. The `initial-ids` parameter preserves the existing behavior of stopping when a newly-created item appears on the stack.

After both changes, `events/game.cljs` will have zero imports from `history/`. The only history import in Game Orchestration will be `core.cljs` calling `history/init-history` during bootstrap and `history-interceptor/register!` — both are wiring concerns appropriate for the app entry point.

## Consequences

- Game Orchestration no longer depends on Practice Platform at the import level — graph matches reality
- The `apply-history-entries` and `make-resolve-entry` helper functions in game.cljs are eliminated
- The history interceptor remains the single authority on history entry management — no duplicated logic
- Resolve-all triggers re-renders between each stack item resolution; this is acceptable for a practice tool and provides visual feedback during storm resolution
- The `resolve-all-handler` INVARIANT comment (warning about keeping batch logic in sync with the interceptor) is eliminated — there is no batch logic to keep in sync
- Opponent turns are a prerequisite for the bot system (Phase 5 in roadmap), so this architectural fix aligns with the product roadmap
