# ADR-009: Decompose Game Loop into Derived State Machine

## Status

Accepted

## Context

yield-impl was a 127-line God function in events/game.cljs that handled priority negotiation, stack resolution, phase advancement, and bot actions in a single conditional chain. When bot turns were introduced (Epic fizzle-ip7s), the function's implicit control flow created integration bugs: the opponent started with no opening hand, bot turns skipped human opponent-turn phase stops, and history entries were missing for bot phase transitions.

A deeper structural issue emerged: yield-impl needed to know whether the human had just cast a spell (to auto-yield after casting) versus explicitly yielding priority. This "cast-and-yield" flag was stored on the selection object (:selection/yield-after?) and lost when selections were cleared during resolution. The flag was an implicit protocol between the selection layer and the game loop — exactly the kind of hidden agreement that ADR-005 warns against.

Additionally, hardcoded :player-1 references in event handlers (cast-spell, resolve-top, play-land, activate-ability, toggle-stop, mulligan) and subscriptions created a landmine: any future bot that casts spells would silently use the human player's identity. The human player's identity was stored in :game/human-player-id during init but never read by event handlers.

## Decision

We will decompose yield-impl into a derived state machine with multimethod dispatch.

**Game loop state machine** (engine/game_loop.cljs): Game loop state (:stack-resolution, :bot-phase, :phase-advancement) is derived from the current game-db — not stored as mutable state. A multimethod dispatches on derived state, so new states add defmethods rather than branches in yield-impl. yield-impl becomes a thin dispatcher that derives state and calls the multimethod.

**Explicit continuation protocol**: Replace the implicit yield-after-cast flag with an explicit continuation protocol. :selection/on-complete stores what should happen after a selection is confirmed. An apply-continuation multimethod dispatches on the continuation type (:yield, :resolve, etc.), making post-selection behavior explicit and extensible. New selection-to-game-loop transitions register continuations, not flags.

**Human player identity**: Store :game/human-player-id in game state and use queries/get-human-player-id in all event handlers and subscriptions instead of hardcoded :player-1. The identity is set once during init and read everywhere else.

## Consequences

- New game loop features (burn bot casting, priority holds, combat phases) add defmethods to the state machine rather than branches in yield-impl
- The continuation protocol makes post-selection behavior explicit — no more implicit flags that can be lost during selection clearing
- Bot casting is safe because event handlers resolve the human player dynamically, not by hardcoded keyword
- yield-impl remains a pure function (derives state, dispatches, returns updated db) — yield-loop still works for auto-resolution
- More indirection: multimethod dispatch and opts maps for event-layer functions require following the dispatch chain to understand behavior
- Game loop state derivation runs on every yield — acceptable cost since derivation is a few predicate checks on game-db
