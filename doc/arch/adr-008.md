# ADR-008: Split Opponent AI out of Practice Platform as independent component

## Status

Accepted

## Context

Practice Platform is declared as owning "history, bots, tactics" — three practice-related features under a single volatility axis ("practice features"). History is implemented (4 files, pure value replacement, never calls engine). Bots and tactics are unimplemented.

An architectural audit surfaced a complection tension: history and bots have fundamentally different interaction patterns with the rest of the system. History is pure — it stores and restores immutable Datascript snapshots without calling the Interpretation Core. Bots will need to heavily query game state via Interpretation Core to make decisions (can I cast this spell? what's in my hand? what targets are available?). These are independent change axes with different dependency patterns.

Additionally, bots are expected to grow significantly in sophistication: goldfish → heuristic bots (burn, clock) → scripted bots (hate pieces, deterministic sequences) → interactive bots (player-directed opponent responses). This growth trajectory suggests high volatility on its own axis, independent of history or tactics changes.

ADR-002 established that "data lives with its interpreter." That principle still holds — bot data (deck configurations, scripts, heuristics) lives with the bot interpretation logic. The change is where that interpreter lives: its own component, not Practice Platform.

## Decision

We will split opponent AI into its own component at the engine layer. Bot decision logic — archetype dispatch, heuristic evaluation, scripted action sequences — is game rule computation: it takes game state and produces action choices. Game Orchestration coordinates when to ask the bot and how to execute its decisions.

The Opponent AI component:
- Owns bot definitions (deck configs, scripts, heuristic rules) — per ADR-002 principle
- Depends on Interpretation Core for game state evaluation (can-cast?, castable cards, valid targets)
- Depends on Data Foundation for game state queries
- Depends on Card Pool for reading card definitions in bot decks
- Is consumed by Game Orchestration, which calls bot logic during opponent turns

Practice Platform retains history and future tactics. Its "pure" characterization (never calls engine) is preserved.

## Consequences

- Bot sophistication grows independently of history — changes to bot heuristics, new archetypes, scripted sequences do not touch history code
- Practice Platform stays pure (its only engine import, history/descriptions → engine/stack, is incidental and could be removed)
- Opponent AI declares its own Interpretation Core dependency explicitly in the graph — no hidden dependencies
- ADR-002's principle ("data lives with interpreter") is preserved — bot data moves with bot logic
- The bot roadmap epics (fizzle-ip7s through fizzle-u07p) now target Opponent AI, not Practice Platform
- Game Orchestration gains a new dependency (Opponent AI) for bot turn coordination
- Refines ADR-002's application: ADR-002 stated "bot definitions live within Practice Platform" — this is superseded for bots; tactics remain in Practice Platform until they materialize
