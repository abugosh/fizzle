# ADR-012: Accept implicit event keyword contracts across component boundaries

## Status

Accepted

## Context

Three components reference re-frame event keywords across component boundaries:

- **Opponent AI** (bots/interceptor.cljs) maintains `bot-trigger-events` — 8 keywords identifying events after which bots evaluate whether to act
- **Practice Platform** (history/interceptor.cljs) maintains `priority-events` — 8 keywords identifying events that represent player priority actions deserving history entries
- **Practice Platform** (history/descriptions.cljs) maintains a description dispatch table — 12 keywords mapped to human-readable action descriptions

The sets overlap but are intentionally distinct: each consumer asks a different semantic question about the shared event vocabulary. Adding a new game event (e.g., `::declare-attackers` for creatures/combat) requires updating each set independently based on its semantic criteria.

ADR-007 accepted implicit value-set contracts for zones, effect types, selection types, and stack-item types. Event keywords are structurally the same pattern — shared keyword vocabulary referenced across components without a central registry. The blast radius of an event keyword rename (3 files) is smaller than schema attribute renames (9+ modules, per ADR-006).

A concern specific to LLM-assisted development: keyword hallucination. An LLM generating code that dispatches `:fizzle.events.game/cast` instead of `::cast-spell` fails silently in re-frame — no handler matches, no error thrown. Human developers rarely invent plausible-but-wrong keywords; LLMs do this regularly. This risk applies equally to all implicit value-sets (ADR-006, ADR-007), not just event keywords.

## Decision

We will keep event keyword contracts implicit, consistent with ADR-007's rationale. Each consumer maintains its own set reflecting its own semantic criteria for which events matter. Centralizing event keywords into a shared registry would braid together independent concerns (what bots care about, what history captures, what gets descriptions) through a shared artifact.

The LLM-hallucination risk is mitigated by the existing test suite: any LLM-generated dispatch to a non-existent event keyword produces no game state change, which test assertions catch. The card testing requirements (CLAUDE.md) mandate full cast/resolve cycle tests through production code, ensuring event dispatch correctness is verified empirically.

## Consequences

- Adding a new game event requires evaluating 3 independent sets based on their semantic criteria — this is intentional, not accidental
- Renaming an event keyword requires grep+replace across 3 files (bots/interceptor.cljs, history/interceptor.cljs, history/descriptions.cljs) plus the defining event handler — same cost as other accepted implicit contracts
- LLM-generated code that dispatches wrong keywords is caught by existing test infrastructure, not by compile-time enforcement
- The independent-set design preserves each consumer's ability to evolve its criteria without coordinating with other consumers
- As the event vocabulary stabilizes (post creatures/combat), the rename risk diminishes further
