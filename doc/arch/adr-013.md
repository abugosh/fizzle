# ADR-013: Builder-declared selection lifecycle

## Status

Accepted

## Context

The selection system's `execute-confirmed-selection` multimethod returns one of three map shapes to signal which post-selection lifecycle path the framework should follow:

- `{:db db}` — standard: framework resumes remaining-effects, cleans up source, applies continuation
- `{:db db :finalized? true}` — finalized: executor handled everything, framework just cleans up app-db
- `{:db db :pending-selection sel}` — chaining: executor needs another round of player input

This protocol works but encodes lifecycle semantics in the executor's return value rather than declaring them upfront. When implementing a new selection type, the developer must understand all three lifecycle paths and choose the correct return shape. Returning the wrong shape causes silent bugs (e.g., double-cleanup, missing grant expiration).

The information needed to determine the lifecycle path is already available when the selection builder runs — builders know whether they're pre-cast (finalized), whether they'll chain (pile-choice present), or whether they're standard (most cases). The lifecycle decision is made implicitly by the executor at execution time, when it could be declared explicitly by the builder at creation time.

## Decision

We will move lifecycle semantics from executor return values to builder-declared metadata on the selection map. Selection builders will set `:selection/lifecycle` to one of `:standard`, `:finalized`, or `:chaining` when constructing the selection state. Executors will return only `{:db db'}` — the game state changes resulting from the player's choices.

`confirm-selection-impl` will read `:selection/lifecycle` from the selection map to determine the post-execution path, rather than inspecting the executor's return shape.

For conditional lifecycle (e.g., tutor that chains to pile-choice only when pile-choice is present), the builder decides at creation time based on the effect data.

`:standard` is the default lifecycle — if `:selection/lifecycle` is not set, the framework assumes standard.

## Consequences

- New selection types default to `:standard` lifecycle — implementors write a builder and executor without understanding lifecycle paths
- Silent lifecycle bugs are eliminated — the framework validates lifecycle metadata rather than inspecting return shapes
- 8 existing executor methods (5 finalized, 3 chaining) need refactoring; 10 standard methods need no changes (returning `{:db}` is already compatible)
- Chaining executors need a mechanism to pass the next selection without returning it (e.g., a well-known key on the selection map that the builder populates)
- Card definitions (EDN data) are unaffected — this is orchestration-internal refactoring
- The engine's `{:needs-selection effect}` tagged return protocol is unaffected
