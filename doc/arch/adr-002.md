# ADR-002: Domain-split data ownership — data lives with its interpreter

## Status

Accepted

## Context

Fizzle uses a data-driven extension model: cards, effects, bots, and tactics are defined as EDN data interpreted by multimethod-based engines. All four share the same extension pattern (EDN + multimethods), which suggests grouping them as a single "Extension Data" component.

However, card EDN is interpreted by `engine/effects.cljs`, bot EDN is interpreted by `bots/`, and tactic EDN is interpreted by `tactics/`. There is no business logic connecting card data to bot data to tactic data — they share a format but have completely separate interpreters. Grouping by format would be functional decomposition (organizing by data type), not volatility decomposition (organizing by independent change axis).

## Decision

We will keep data definitions co-located with their interpretation engines. Card definitions live in Card Pool (separate component, highest volatility). Bot and tactic definitions live within Practice Platform alongside their respective interpretation logic. Each domain owns its data and its interpreter as a single volatility axis.

If data loading moves to external sources in the future, a Data Gateway component at the Resource Accessor layer would sit beneath Card Pool and Practice Platform, separating "how data enters" from "what data means."

## Consequences

- Adding a new card touches Card Pool (data) + Interpretation Core (if new effect type needed) — two components maximum
- Adding a new bot behavior touches only Practice Platform — one component
- Adding a new tactic scenario touches only Practice Platform — one component
- Card Pool has the highest volatility and is the smallest component — changes are fast and isolated
- If card, bot, and tactic data diverge in structure over time, they can evolve independently
- If data loading moves external, the Data Gateway decision is deferred until the force materializes, avoiding premature abstraction
